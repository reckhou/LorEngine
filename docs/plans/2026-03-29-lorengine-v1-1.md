# Lorengine v1.1 Implementation Plan

**Goal:** Add collaborative editing (Decap CMS), hierarchical page organization (tree view), user-friendly API key handling, page references with backlinks, and release tooling. All decisions from 2026-03-29 design session.

**Architecture:** Build script enhanced to compute hierarchy metadata and backlinks from folder structure. Wiki frontend renders collapsible tree sidebar from hierarchy. API key modal persists to localStorage with delete option. Decap CMS adds `/admin` interface for WYSIWYG editing with image upload, committing back to repo. release-please automates versioning from conventional commits.

**Tech Stack:** Python 3.11+ stdlib (build script), vanilla JS ES2020+ (sidebar/modal), Toast UI Editor (WYSIWYG via CDN), Decap CMS (headless CMS via CDN), release-please GitHub Action.

## Task 1: Build script v1.1 — hierarchy, backlinks, branding, config validation

**Files:** `build_wiki.py`, `config.yml`

Add three core functions to build_wiki.py:

1. **compute_hierarchy(entries)** — Build parent/child tree from folder paths. For `docs/combat/attacks.md`, parent is `docs/combat.md`. Each entry gains `parent` field (null for top-level) and `children` list.

2. **compute_backlinks(entries)** — Scan all doc bodies for markdown links `[text](docs/...)` and populate `backlinks` list on each entry showing which docs reference it.

3. **validate_config(parsed, defaults)** — Warn on unknown top-level config keys (non-fatal) to catch typos in config.yml.

Update `load_config()` to add `branding` section with defaults for `favicon` and `logo`.

Update `assemble_site()` to:
- Copy `docs/assets/` into `_site/assets/` (for in-repo images)
- Inject favicon/logo paths into template replacements

Update `main()` call sequence:
```
search_index = build_search_index(root)
search_index = compute_hierarchy(search_index)
search_index = compute_backlinks(search_index)
```

**Verify:** Run build and check:
- `search_index.json` has `parent`, `children`, `backlinks` fields on each entry
- Existing docs work (example-doc.md has empty hierarchy/backlinks)
- Config warning appears for unknown keys
- Branding values injected into HTML

## Task 2: Tree sidebar — folder-based tree with filter/sort and persistence

**Files:** `wiki/wiki.js`, `wiki/style.css`, `wiki/index.html`, `wiki/page.html`, `wiki/search.html`

**Depends on:** Task 1

Render collapsible tree from hierarchy data with filter and sort controls.

**JS functions:**
- `buildDocTree(data)` — Create tree nodes from entries with parent/child links. Store expanded state from localStorage.
- `renderTreeNode(node, nodes, filterText, sortOrder)` — Recursively render tree item with expand/collapse toggle. When filtering, hide non-matching docs but keep unmatched parents visible (preserve structure).
- `initTreeSidebar(data, container)` — Set up tree UI with filter input and sort dropdown. Save expanded state on toggle.

**CSS:** Add styles for `.tree-sidebar`, `.tree-toggle` (with rotation animation), `.tree-link`, `.tree-filter`, `.tree-sort`, `.tree-children`.

**HTML:** Add tree sidebar to all three pages:
```
<aside class="page-sidebar">
  <div class="tree-sidebar">
    <div class="tree-controls">
      <input type="text" class="tree-filter" placeholder="Filter pages...">
      <select class="tree-sort">
        <option value="alpha-asc">A-Z</option>
        <option value="alpha-desc">Z-A</option>
        <option value="date-asc">Oldest first</option>
        <option value="date-desc">Newest first</option>
      </select>
    </div>
    <ul class="tree-list"></ul>
  </div>
</aside>
```

**Verify:**
- Tree shows all docs with expand/collapse arrows
- Filter hides non-matching but keeps parents visible
- Sort by alpha/date works
- Expanded state persists on page navigation
- Mobile: sidebar hidden by default with toggle button

## Task 3: API key modal system — blocking modal, remember/delete

**Files:** `wiki/wiki.js`, `wiki/style.css`, `wiki/page.html`

Implement blocking modal for API key entry with remember/delete options.

**JS functions:**
- `showAPIKeyModal(callback)` — Show modal with password input, remember checkbox, submit button. Modal is fixed position, centered, blocks page interaction. Call callback(key, remember) on submit.
- Update `initAISidebar()` — Check localStorage for saved key. If missing, show modal. If present, use it silently.
- `initSidebarUI()` — If key was remembered, add delete button that removes it from localStorage.

**CSS:** Modal overlay styles, input styling, delete button styling with warning icon.

**Verify:**
- Fresh page (no saved key) → blocking modal appears immediately
- Enter key, uncheck remember → sidebar works, refresh shows modal again
- Enter key, check remember → sidebar works, delete button visible, refresh keeps key
- Click delete → key removed, refresh shows modal
- Key never appears in network requests (only sent to API or Cloudflare proxy)

## Task 4: Backlinks + page references — display backlinks, title autocomplete

**Files:** `wiki/wiki.js`, `wiki/style.css`, `wiki/page.html`

**Depends on:** Task 1

Display pages that reference the current doc. Add title-based autocomplete for creating links.

**JS functions:**
- `renderBacklinks(backlinks, searchIndex)` — Render section listing docs that link to this one. Uses backlinks array from search index (populated by Task 1).
- `buildTitleAutocomplete(searchIndex)` — Create Map of title (lowercased) → doc id.
- `autocompleteTitle(input, titleMap)` — Return matching titles for partial input (case-insensitive).

**CSS:** Styles for `.backlinks` section with border separator, link styling.

**HTML:** Add to page.html:
```
<div id="page-backlinks"></div>
```

In initialization, after rendering doc body, render backlinks if they exist.

**Verify:**
- Create docs: a.md links to b.md via standard markdown links
- View b.md → "Referenced by" section shows a.md
- No section if doc has no backlinks
- Links are clickable

## Task 6: Release tooling — release-please, GitHub Actions, semver

**Files:** `.github/workflows/release.yml` (new), `CHANGELOG.md` (new)

**Independent** — can be done anytime.

Automate versioning and changelog from conventional commits using release-please.

**.github/workflows/release.yml:**
- Trigger: push to main
- Action: release-please (googleapis/release-please-action)
- Reads commits, bumps version (feat→minor, fix→patch, feat!→major)
- Updates CHANGELOG.md
- Creates GitHub Release
- Tags commit with semver (v1.1.0, etc.)

**CHANGELOG.md template:**
- Structure per Keep a Changelog
- release-please populates "Unreleased" section, then versions it on release

**Update README.md:**
- Document conventional commit format
- Note that forks can pin to specific versions via git tags

**Verify:**
- Merge `feat: add X` → release-please PR opens
- Merge release-please PR → v1.1.0 tag created, GitHub Release published
- Downstream forks can `git checkout v1.0.0`
